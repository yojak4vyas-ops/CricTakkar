## FOUR-SOURCE VERIFICATION RULE — PERMANENT (Added Day 16)

This is the master fact-checking standard for the entire app. It applies to every question, fact, and data point across ALL content areas: Daily Quiz + Category Quiz — IPL/Test/ODI/T20 (all now live in one central, category-tagged file, `question-bank.js`, as of Day 17), On This Day (`onthisday.js`), Ranking Challenge (`ranking.js`), Cricket Poetry — poems and Poetry Quiz (`poems.js`), and Cricket Wordle (`players.js`). It supersedes the older 2-source ("ESPNcricinfo + Wikipedia") wording used elsewhere in this file — wherever this file says to check 2 sources, this 4-source rule now takes precedence.

STANDING SCOPE — THIS RULE IS NOT LIMITED TO THE LIST ABOVE: it automatically applies to any new quiz mode, category, sub-category, game, or content area added to CricTakkar at any point in the future (e.g. a future T20 World Cup category, a new Phase 2/3 feature with cricket facts, a new player database for another game mode) — Claude does not need a separate instruction to apply it each time. It also applies to any additional data the user or Claude adds to an existing area later — more questions added to any quiz, more players added to Wordle, more events added to On This Day, more challenges added to Ranking, more poems added to Poetry. The rule runs by default on all cricket facts/data in this codebase, existing or future, unless the user explicitly says otherwise for a specific case.

THE RULE:
- Every fact must be checked against all 4 of: **the Cricbuzz website (cricbuzz.com), ESPNcricinfo, Wikipedia, and ICC official records** (icc-cricket.com). (Wording clarified Day 28: "Cricbuzz" throughout this file specifically means the cricbuzz.com website — see the KNOWN TOOL LIMITATION note below on why it's flagged unreachable every time.)
- MECHANISM — NOT OPTIONAL (added Day 17, session 6, after a real lapse): "checked against" means an explicit, separate, domain-restricted search per site — using WebSearch's `allowed_domains` parameter set to exactly one of espncricinfo.com / wikipedia.org / icc-cricket.com / cricbuzz.com per query — for every single fact. A broad, unrestricted web search that happens to cite some of these sites in its results is NOT the same thing and does NOT satisfy this rule, even if the fact looks consistent across what came back. The lapse that triggered this clause: an entire batch of 30 questions (Day 17, session 5) was "verified" using broad searches instead of per-site ones — the facts turned out fine on a spot re-check, but the process itself didn't match what this rule requires, and the user caught it, not Claude. Default to the strictest reading of this rule, always — it is described as non-negotiable and hardcore by the user, so ambiguity gets resolved toward more rigor, never toward what's faster.
- This is an AND, not an OR. All four sources must show the exact same fact/number/detail.
- Zero tolerance: ANY mismatch between the sources, however small (e.g. 703 vs 704 wickets, "left-arm orthodox" vs "left-arm unorthodox", one team missing from an IPL list), counts as a mismatch.
- Claude never resolves a mismatch on its own — no picking the "more likely" source, no majority rule, no judgment call. Every mismatch gets flagged to the user instead.
- Flags are collected together and presented as one list at the end of a batch/file, not one at a time as they come up.
- The user decides the resolution for each flagged item (fix it a specific way, drop it, or keep with a caveat comment). Claude waits for that decision before writing the data.
- Applies to BOTH the existing database (a full re-verification pass, started Day 16, working through content areas one at a time) and every new question/fact/player added from now on — nothing new goes in without clearing all 4 sources first.

KNOWN TOOL LIMITATION (found Day 16): Claude's web tools (WebSearch/WebFetch) cannot access **the Cricbuzz website, cricbuzz.com specifically** — it blocks the crawler on both domain-filtered search and direct fetch. (This is specifically about the cricbuzz.com website being blocked to Claude's tools — not a statement about Cricbuzz as a brand/source being unreliable.) Per user decision: Cricbuzz stays in the rule, and every single time it is unreachable, Claude must explicitly flag that fact as "Cricbuzz website unreachable" alongside the verification result — this is a standing, permanent flag, not a one-time note. Verification proceeds on the other 3 sources (ESPNcricinfo + Wikipedia + ICC) when the Cricbuzz website can't be reached, but the gap must always be visible to the user, never silently skipped.

KNOWN TOOL LIMITATION, PART 2 (found Day 18, IPL Batch 2): direct WebFetch of iplt20.com pages (stats pages, points tables) returns 403 Forbidden, same failure mode as Cricbuzz — and the same is true of direct WebFetch on espncricinfo.com stat pages specifically (search-based access to espncricinfo.com still works fine). WebSearch with `allowed_domains` set to iplt20.com DOES still return usable results, but only when the query is phrased with the specific expected fact/number already in it — vague queries return only page titles/links with no real content in the AI-synthesized summary. So for IPL facts: run the targeted iplt20.com WebSearch query, and if it doesn't confirm the number, don't treat that as a real source mismatch — retry with a more specific query before concluding anything. Separately, and unrelated to reachability: WebSearch's own AI-synthesized answer text has repeatedly proven unreliable on exact numbers even when the underlying source page is reachable (e.g. it misquoted Bhuvneshwar Kumar's IPL 2017 wicket count as 23 instead of 26, apparently cross-contaminating it with his 2016 tally, during Batch 2 verification) — always cross-check a synthesized number against a second, independent query before trusting it.

SCOPE AT TIME OF WRITING (Day 16): ~155 existing items across Daily Quiz, Category Quiz (4 categories), On This Day, and Ranking Challenge, plus 130 Wordle players × 8 attributes each, all need to be re-verified under this standard. See Current Build Status for progress through the backlog.

## ADDING NEW CONTENT — STANDARD OPERATING PROCEDURE (Added Day 17, session 5)

This is the permanent, repeatable process Claude follows any time the user says "add questions" / "add players" / "add [X]" to any content area — the user should never need to re-explain this process. It applies to all 5 content files: `question-bank.js` (quiz questions), `players.js` (Wordle), `onthisday.js` (dated events), `ranking.js` (challenges), `poems.js` (poems + Poetry Quiz).

**Step 1 — Figure out the target file and shape.** Match the request to the right file (see the field shapes documented earlier in this file for each). If the user gives a topic/player/theme rather than full content, Claude drafts the actual question/entry itself.

**Step 2 — Check for duplicates.** Before writing anything, check the existing bank for the same fact already covered under different phrasing (this is what caused the Day 17 merge cleanup). Skip or merge instead of adding a near-duplicate.

**Step 3 — Verify every fact.** Run the Four-Source Verification Rule above on every new fact — live web search, not training data. Cricbuzz is expected to be unreachable (standing tool limitation) — flag it every time, don't silently skip it. If a fact is a "current record" / "most X" / "reigning champion" type that can change over time (Question Quality Rule), handle it explicitly: note in the fact text if the record is still growing (an active player), confirm if the record is now closed (a retired player), and phrase dated tournament results around their specific year rather than "current champion" so the question never goes stale.

**Step 4 — Tag correctly.** For `question-bank.js`: assign `category` (ipl/test/odi/t20/general) and `era` (Pre-1970s/1970s-80s/1990s/2000s/2010s/2020s/General) per the existing tagging rules. For other files, follow that file's existing field schema exactly (e.g. Wordle's 8 attributes, Ranking's 5-item ordered list).

**Step 5 — Structural validation.** Before considering the content done, run a quick integrity check confirming every new entry has all required fields, exactly 4 options where applicable, and a valid correct-answer index — this catches typos/missing fields before they ever reach a user.

**Step 6 — Test live in the browser.** Load the relevant page(s), confirm the new content renders correctly with no console errors, and confirm any live counts (home page stat strip, category pool sizes) updated correctly with zero manual editing needed elsewhere.

**Step 7 — Report back plainly.** Give the user the full list of what was added (not just a count), any facts that needed special "growing record" handling, and explicitly confirm the verification pass ran (which sources, Cricbuzz status) rather than just asserting "verified" — this is what lets the user actually check the work, not just trust it blindly.

**Step 8 — Commit, push, deploy, and update Current Build Status** — per the standing project rule, automatically, without being asked.

If at any point a fact can't be confirmed clean across the reachable sources, it gets flagged to the user per the Four-Source Verification Rule's mismatch-handling process — Claude never guesses or picks a "likely" answer to keep moving.

## CRICKET WORDLE PLAYER DATA RULE — PERMANENT (Added Day 10, updated Day 14)

The Cricket Wordle game uses a player database in `players.js`. Every player entry has 8 guessable attributes: country, role, battingStyle, bowlingStyle, debutYear, format, iplTeams, iccTrophies.

- `bowlingStyle` — added Day 14. Mark "NA" only if the player has a genuinely confirmed zero-ball bowling record (e.g. Rishabh Pant, Sanju Samson — explicitly 0 overs/wickets in their career stats). If they have bowled even a single delivery anywhere in their career — even just once, even a part-timer's over — record their actual bowling style instead of NA. Caught this the hard way on Day 14: MS Dhoni was wrongly marked NA despite having bowled in internationals (took 1 wicket, 2009 Champions Trophy, right-arm medium). If no source can confirm bowling record either way (bowled or never bowled), mark NA but add a comment flagging it as unconfirmed rather than proven, so it can be revisited (see KL Rahul's entry as the template for this).
- `iplTeams` — added Day 14, replaces the old single `iplTeam` + `iplTeamsCount` fields. An array of every IPL team the player has actually played a match for, in chronological order — not just their current or most recent team. A team they were signed to but never played a game for (e.g. Axar Patel's Mumbai Indians stint) does not count. Empty array `[]` for players who never played IPL.
- IPL TEAM LIST SEARCH METHOD — PERMANENT (added Day 14, after Batch 2 needed a second correction pass): a single combined search query (e.g. "Player X batting bowling style IPL teams") reliably surfaces batting/bowling style but is NOT reliable for the full IPL team list — it tends to surface only the player's most recent or most prominent team and silently miss earlier ones. This caused 6 real misses in Batch 2 alone (Guptill, Ponting, Gilchrist, Russell, Chanderpaul, Ross Taylor), including two players wrongly marked "Didn't play IPL" who had actually played for multiple franchises. Always run a SEPARATE, dedicated query for IPL history specifically — phrase it as "complete IPL career all teams" or "did X ever play IPL" rather than folding it into the combined batting/bowling query — for every player, not just the ones that seem like they might have a complicated history. This is now mandatory before marking any player's `iplTeams` as a single team or empty array.
- CROSS-VERIFICATION STANDARD: every one of the 8 attributes (not just batting/bowling style and IPL teams) must pass the FOUR-SOURCE VERIFICATION RULE above — Cricbuzz + ESPNcricinfo + Wikipedia + ICC official records, zero-tolerance AND logic. Any mismatch is flagged to the user, collected together per batch, never resolved silently.
- `pickTodayPlayer()` in wordle.js only selects the daily mystery player from players that have `bowlingStyle` defined (i.e., have been through this verification pass) — this expands automatically as each country batch gets done, so nobody is ever shown as the day's answer with unverified data.

FORMAT RULE — THIS IS CRITICAL:
- The `format` field must show the EXACT combination of official international formats a player has played — never a lossy catch-all. There are 7 valid values: `Test`, `ODI`, `T20` (single format only), `Test+ODI`, `Test+T20`, `ODI+T20` (exactly two formats), and `All-format` (all three).
- "Playing" a format means at least 1 official international match: a Test, an ODI, or a T20I.
- IPL, Big Bash, domestic T20, franchise cricket, ICL — none of these count as T20I (or any international format). Only bilateral/ICC-event international caps count.
- If a player retired before T20 Internationals existed (before Feb 2005), they cannot have a T20I credit — full stop, no need to even check.
- **PERMANENT FIX (Day 26, session 3):** the field used to be a lossy binary — "All-format" (all 3) vs "ODI" (used as a catch-all for "not all 3", regardless of which 2 formats, or even just 1, a player actually played). A user-flagged bug (Yuzvendra Chahal showing "ODI" despite having played ODI+T20I) triggered a full audit of every player marked "ODI" — 95 players were reclassified using the 7-value scheme above. The audit also caught 2 genuine pre-existing errors unrelated to the scheme change: Chaminda Vaas and Inzamam-ul-Haq had each actually played 1 T20I (confirmed via ESPNcricinfo/Wikipedia) and are now correctly `All-format`, not `Test+ODI`. It also caught 2 players who had zero ODI caps despite being labelled "ODI" (Devdutt Padikkal, Dhruv Jurel — both actually `Test+T20`), and 6 players with zero ODI *and* zero T20I caps who were mislabelled "ODI" instead of `Test` (Manav Suthar, Anshul Kamboj, Akash Deep, Sarfaraz Khan, Shahbaz Nadeem, KS Bharat). `wordle.js`'s `comparePlayer()` was rewritten to compare formats by set-overlap (green = identical combination, yellow = share at least one format without matching exactly, grey = no shared format) instead of a hardcoded "All-format is special" rule — this generalizes correctly to any of the 7 values.
- **One player (Harsh Dubey) stayed flagged rather than resolved:** he received a T20I squad call-up for India's July 2026 Zimbabwe tour, but an actual match cap could not be confirmed at time of writing — kept as `ODI` with a comment flag rather than asserted as `ODI+T20`, same treatment as other unconfirmed-attribute cases (see KL Rahul precedent below).

VERIFIED FORMAT REFERENCE EXAMPLES:
- Brian Lara → Test+ODI (retired 2007, zero T20Is — Southern Rocks T20 in 2010 was domestic Zimbabwe cricket)
- Shane Warne → Test+ODI (retired January 2007, zero T20Is — IPL was franchise cricket not T20I)
- Wasim Akram → Test+ODI (retired 2003, T20I format did not exist)
- Kapil Dev → Test+ODI (retired 1994, T20I format did not exist)
- Sourav Ganguly → Test+ODI (zero T20 Internationals for India — all T20 appearances were IPL/domestic only)
- Rahul Dravid → All-format (played 1 T20I vs England in 2011 — verified on ESPNcricinfo and Britannica)
- Yuzvendra Chahal → ODI+T20 (ODI debut and T20I debut both June 2016, zero Test caps)
- Devdutt Padikkal → Test+T20 (Test debut 2024, T20I debut 2021, explicitly confirmed zero ODI caps)

BEFORE ADDING ANY NEW PLAYER TO players.js:
- Always verify all 3 formats on ESPNcricinfo first — check the Debut/Last Matches page for the Test, ODI, AND T20I sections individually. Do not stop checking once you find one or two — confirm the presence or explicit absence of all three before picking the format value.
- Never assume a player is All-format (or any particular combination) because they played IPL, Big Bash, or other domestic/franchise T20 cricket.
- Always add a comment line with the verification source and match counts per format.

## RANKING CHALLENGE EXPANSION PROJECT — PERMANENT (Added Day 27, redesigned same day)

Started because Ranking Challenge (`ranking.js`) had sat at exactly 5 fixed challenges since Day 11 — every session showed the same 5, in the same order, no pool to draw from (unlike Quiz's 330+ question bank or Wordle's 255-player database). Day 27 restructured this twice in one day: first into a simple fixed pool (superseded), then into the growing-leaderboard design below after further discussion with the user about how the game should actually feel.

**HOW THE GAME WORKS NOW:** a play session shows **4 challenges** (changed from 5). Each challenge is a "rank these 5 players by X" question. Two sources feed a session:
- **`RANKING_PARAMETERS`** (in `ranking-challenges.js`) — growing "leaderboard" topics like "Test batting average." Each parameter holds a `leaders` array of players sorted best-to-worst on that stat. At session start, the engine randomly draws a 5-player **window** from that list — see the WINDOW RULE below. Target size per parameter: **25 leaders** (a long-term target, grown gradually over many future sessions — not expected today or this week).
- **`FIXED_CHALLENGES`** — small one-off topics that don't fit a growing sorted leaderboard (IPL title ties, "who debuted earliest," "highest individual score" as a standing record). Stay as a hand-picked set of exactly 5 for now.

**WINDOW RULE — the core new mechanic:** when a parameter's `leaders` list has grown past 5, the game doesn't always show the same top 5 — it randomly picks a 5-player window from anywhere in the list, with one constraint: **consecutive picked players (sorted by their rank in that list) can never be more than 3 ranks apart.** This keeps every generated challenge a plausible, tightly-clustered comparison (e.g. ranks 6, 8, 9, 11, 14 is fine — gaps of 2,1,2,3; ranks 1, 2, 3, 20, 21 is not, gap of 17 is too big) instead of jumping from an obvious #1 to an obscure #20. Because the leaders list is already sorted, the correct answer for any drawn window is just that window's own order — nothing needs to be hand-written per instance. Implemented as `pickWindowIndices()` in `ranking.js` (deterministic construction, not retry-based — an earlier retry-based version had a real bug where it could randomly fail ~10-20% of the time on a freshly-started 5-player list and silently drop that topic from the session; fixed same day, verified with 2,200+ trials at list sizes 5/12/25 with zero failures).

**STALENESS RULE — permanent, applies to every leader added to any parameter:** for career-cumulative stats (Batting Average, Most Runs, Strike Rate, Most Centuries, Bowling Average, Most Wickets, Economy Rate, Most Five-Wicket Hauls, Most Maiden Overs, all Fielding/WK stats, Most Matches Played, Most Wins as Captain, etc.) — only fully retired players may be used. An active player's number keeps moving, which could silently flip a ranking's correct order without anyone noticing — worse than a single stale quiz fact, since a ranking's correctness depends on the relative order of all 5. One-off record stats (Highest Individual Score, Best Bowling Figures, Most Catches in an Innings/Match) may use an active current record-holder, same as the Question Quality Rule's "growing record" handling elsewhere — flag it as a still-standing record in a comment.

**IF A CELL CAN'T FIND ENOUGH RETIRED QUALIFIERS:** T20I and IPL skew toward active players since both are younger/shorter-career formats. If a cell can't find enough clearly-retired players who meaningfully qualify (a reasonable minimum threshold, e.g. min innings/wickets/matches so the list isn't skewed by tiny sample sizes), don't stretch to include an active player in a cumulative-stat parameter. Either use a smaller qualifying pool if still meaningfully accurate, skip the cell for now and flag it, or reframe as a dated/closed record instead.

**THE FULL TOPIC TAXONOMY (locked in Day 27):** four top-level buckets, each with sub-parameters, each sub-parameter targeted across all 4 formats (Test / ODI / T20I / IPL) where the stat sensibly applies:

- **BATTING** (8 sub-parameters): Batting Average, Most Runs, Strike Rate, Most Centuries, Highest Individual Score (one-off record), Most Fifties, Most Sixes, Most Fours
- **BOWLING** (7 sub-parameters): Bowling Average, Most Wickets, Economy Rate, Best Bowling Figures (one-off record), Most Five-Wicket Hauls, Bowling Strike Rate, Most Maiden Overs
- **FIELDING** (incl. wicket-keeping) (7 sub-parameters): Most Catches (fielder, non-keeper), Most Dismissals (wicketkeeper — catches + stumpings), Most Stumpings, Most Catches (wicketkeeper only), Most Run-outs Effected, Most Catches in a Single Innings (one-off record), Most Catches in a Single Match (one-off record)
- **OTHERS** (7 sub-parameters): Earliest Debut, Most Trophies/Titles Won, Most Matches Played, Most Wins as Captain, Most Player-of-the-Match Awards, Most Player-of-the-Series/Tournament Awards, Youngest Debutant

That's 29 sub-parameters. Not every sub-parameter × format combination will turn out to make sense (e.g. maiden overs are vanishingly rare in T20I/IPL) — rather than pre-deciding every exclusion now, a cell that turns out non-sensical or unresearchable gets skipped and flagged when a session actually reaches it, same handling as the "can't find enough retired qualifiers" case above. Realistic total once non-sensical combinations are excluded: well under the theoretical 29×4=116 ceiling, but still a large, multi-month project on the scale of the Wordle cap-expansion work.
